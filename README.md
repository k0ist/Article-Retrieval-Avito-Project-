# Гибридный RAG поиск — двухэтапный поиск релевантных статей (MAP@10)

Решение задачи поиска релевантных статей справки по пользовательскому запросу.
Для каждого запроса возвращается ранжированный список из до 10 `article_id`,
отсортированный по убыванию релевантности.

## Архитектура

Пайплайн состоит из двух этапов:

**1. Retrieval (широкий поиск кандидатов)**
- **BM25** — лексический поиск по стеммированным токенам (`nltk` +
  `SnowballStemmer("russian")`), работает по полю `title` (с утроенным весом) + `body`.
- **Dense embeddings** — семантический поиск через `intfloat/multilingual-e5-large`
  и `faiss` (`IndexFlatIP`), работает по укороченному тексту (заголовок + первые
  150 слов текста статьи).
- Результаты BM25 и embeddings объединяются через **Reciprocal Rank Fusion (RRF)**.

**2. Reranking (точное ранжирование для улучшения MAP@10)**
- Top-50 кандидатов после RRF пересортировываются **cross-encoder**'ом
  (`BAAI/bge-reranker-v2-m3`), который оценивает пару (запрос, текст статьи)
  целиком, а не по отдельным векторам — это точнее bi-encoder подхода,
  но дороже по вычислениям, поэтому применяется только к узкому пулу кандидатов.
- Из пересортированного списка берутся top-10.
- Reranking всех запросов выполняется **одним батчевым вызовом**: все пары
  (запрос, текст статьи) по всем запросам собираются в единый плоский список
  и прогоняются через `reranker.predict()` разом, вместо цикла с отдельным
  вызовом на каждый запрос — это существенно эффективнее использует GPU.

## Обработка HTML статей

`body` в `articles.f` хранится как HTML с таблицами, вкладками и спойлерами,
поэтому перед индексацией текст очищается:

1. **Парсинг через BeautifulSoup**, удаление `<script>` и `<style>` тегов —
   их содержимое не несёт смысловой информации и только шумит в индексе.
2. **Извлечение текста** через `get_text(separator=" ")` — теги разметки
   убираются, остаётся plain text; множественные пробелы/переносы схлопываются
   в один пробел (`re.sub(r"\s+", " ", text)`).
3. **Два разных представления текста статьи** готовятся отдельно под разные
   компоненты retrieval, так как у них разные требования к длине и токенизации:
   - `bm25_text` — заголовок (утроенный вес, т.к. он самый информативный
     сигнал) + весь очищенный `body` целиком;
   - `nn_text` — заголовок + первые 150 слов `body`. Обрезка нужна, потому что
     модель эмбеддингов (`multilingual-e5-large`) имеет ограничение на длину
     контекста, а длинные статьи справки часто содержат много деталей ближе к
     концу, которые для семантического поиска по существу вопроса не так важны,
     как первый абзац с сутью.
4. Отдельно токены для BM25 дополнительно проводим через стмминг, чтобы словоформы вроде "кроссовки"/"кроссовок"
   считались одним токеном — для эмбеддингов это не требуется, так как модель
   уже учитывает это.

## Признаки, модели и алгоритмы

| Компонент | Что используется | Зачем |
|---|---|---|
| Лексический поиск | BM25 (`rank_bm25`) по стеммированным токенам | ловит точные совпадения терминов, брендов, названий |
| Семантический поиск | `intfloat/multilingual-e5-large` + FAISS (`IndexFlatIP`) | ловит перефразированные и смысловые совпадения без точного пересечения слов |
| Объединение ранжирований | Reciprocal Rank Fusion (RRF) | сила каждого метода компенсирует слабости другого без ручной калибровки шкал score |
| Точное ранжирование | Cross-encoder `BAAI/bge-reranker-v2-m3` | оценивает пару (запрос, документ) целиком — точнее bi-encoder подхода на топ-кандидатах |

## Структура данных

| Файл | Описание |
|---|---|
| `articles.f` | база статей: `article_id`, `title`, `body` (HTML) |
| `calibration.f` | 500 размеченных запросов для локальной проверки: `query_id`, `query_text`, `ground_truth` |
| `test.f` | 500 запросов без разметки: `query_id`, `query_text` |
| `answer.csv` | результат: `query_id`, `answer` (id статей через пробел, топ-10, порядок важен) |

## Метрика

Решение оценивается по **MAP@10** — среднему значению AP@10 по всем запросам.
AP@10 оценивает не только сам факт нахождения
релевантного документа в топ-10, но и то, на какой позиции он найден: чем выше - тем лучше.

## Как проверялось качество

`calibration.f` использовался как локальный полигон для двух целей:

- **Сравнение с baseline.** Отдельно был прогнан вариант только с BM25
  (top-10 напрямую от лексического поиска, без embeddings и без reranker'а),
  чтобы понять, какой реальный прирост даёт гибридная схема и оправдана ли
  её дополнительная вычислительная стоимость.
- **Подбор гиперпараметров.** На calibration.f тюнились: вес заголовка в
  `bm25_text`, размер пула кандидатов после RRF (`candidate_k`), параметр `k`
  самого RRF, и то, включать ли reranker вообще — та комбинация, что даёт
  максимальный MAP@10 на калибровке, используется для test.f.

| Вариант пайплайна | MAP@10 на calibration.f |
|---|---|
| Только BM25 | 0.2588 |
| BM25 + embeddings (RRF, без reranker'а) | 0.3835 |
| Полный гибрид (BM25 + embeddings + reranker) | 0.39 |


## Методы борьбы с ошибками

- Для борьбы с потерей структуры таблиц предусмотрен вариант очистки HTML,
  где строки таблиц разворачиваются как `"ячейка1 | ячейка2 | ..."`, а не
  просто вырезаются как обычный текст.
- Reranking введён отдельным этапом именно потому, что BM25 и bi-encoder
  эмбеддинги в равной мере уязвимы к "семантически похожим, но неверным"
  документам — cross-encoder, видящий запрос и документ одновременно, лучше
  различает такие случаи.
- Батчевый вызов reranker'а (все пары сразу, один `predict()` вместо цикла)
  введён для сокращения времени выполнения на GPU без изменения итоговых
  предсказаний и метрики.

## Используемые модели

Embedding -  `intfloat/multilingual-e5-large`
Reranker - `BAAI/bge-reranker-v2-m3`
